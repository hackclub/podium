# Database

PostgreSQL with async SQLModel ORM. Models live in `backend/podium/db/postgres/`.

## Query Patterns

Use `session.get()` for primary key lookups:
```python
event = await session.get(Event, event_id)
```

Use typed helpers for other queries:
```python
from podium.db.postgres import scalar_one_or_none, scalar_all

event = await scalar_one_or_none(session, select(Event).where(Event.slug == slug))
projects = await scalar_all(session, select(Project).where(Project.event_id == id))
```

Create models with `model_validate`:
```python
new_event = Event.model_validate(event_create, update={"owner_id": user.id})
session.add(new_event)
await session.commit()
```

Update models with `sqlmodel_update`:
```python
event.sqlmodel_update(event_update.model_dump(exclude_unset=True, exclude_none=True))
await session.commit()
```

## Eager Loading

Without eager loading, accessing a relationship triggers a separate query (N+1 problem). Use `selectinload` to fetch related records in one query:

```python
# Bad: each project.votes access triggers a query
projects = await scalar_all(session, select(Project))
for p in projects:
    print(p.votes)  # N additional queries

# Good: votes loaded upfront with one extra query
projects = await scalar_all(
    session,
    select(Project).options(selectinload(Project.votes))
)
```

See [SQLAlchemy loading docs](https://docs.sqlalchemy.org/en/20/orm/queryguide/relationships.html#relationship-loading-techniques).

## Computed Fields

Use `@computed_field` for derived values from relationships. Must eager-load the relationship first:

```python
# Model
@computed_field
@property
def points(self) -> int:
    return len(self.votes)

# Query - must include selectinload
stmt = select(Project).options(selectinload(Project.votes))
```

See [Pydantic computed_field docs](https://docs.pydantic.dev/latest/concepts/fields/#computed-fields).

## Read Replicas

In production, read-heavy public endpoints use a separate read-only database connection to offload the primary. In dev (single Postgres instance), both connections point to the same database.

**Use `get_ro_session` for read-only GET endpoints:**
```python
from podium.db.postgres import get_ro_session

@router.get("/events/official")
async def list_official_events(
    session: Annotated[AsyncSession, Depends(get_ro_session)],
) -> list[EventPublic]:
    ...
```

**Use `get_session` for any endpoint that writes data** (POST, PUT, DELETE).

**Config:** Set `PODIUM_DATABASE_URL_RO` in Doppler. If not set, both `get_session` and `get_ro_session` use the same `PODIUM_DATABASE_URL` — zero behavior change in dev.

## Migrations

Alembic-managed. Files in `backend/migrations/versions/` form a linked list via `down_revision`. Postgres tracks the current head in an `alembic_version` table.

```bash
# Apply pending migrations
doppler run --config dev -- uv run alembic upgrade head

# Generate a new migration from model changes
doppler run --config dev -- uv run alembic revision --autogenerate -m "description"

# Roll back one step
doppler run --config dev -- uv run alembic downgrade -1
```

**Writing a migration — always review autogenerated output.** Autogenerate diffs SQLModel metadata against the live DB and emits `op.add_column` / `op.drop_column` / etc. It **cannot infer data transforms**. If you're renaming a column or replacing old fields with a new one, hand-edit the file to backfill data *before* dropping the old columns. See `a1b2c3d4e5f6_add_event_phase_*.py` for the pattern: add new column → `op.execute("UPDATE ...")` to backfill → drop old columns.

**Branch conflict.** If two PRs create migrations from the same parent you'll have two heads. Fix by editing the second migration's `down_revision` to point at the first.

## Reset

```bash
./scripts/reset-migrate.sh                 # Reset local DB (empty)
./scripts/reset-migrate.sh --sync <URL>    # Sync schema + data from another Postgres
```

## Backups

Manual, not scheduled. See [backups/README.md](../backups/README.md).

```bash
./scripts/backup-db.sh              # Dev DB → backups/*.dump + CSV
./scripts/backup-db.sh --config prd # Production DB
```

`.dump` is the restorable artifact (`pg_restore --clean --if-exists`). CSVs are for inspection only.

## Caching

Redis-backed, optional. Module: `backend/podium/cache/`. Helpers: `cache_get`, `cache_set`, `cache_delete`.

- **Graceful degradation:** if `PODIUM_REDIS_URL` is unset or Redis is unreachable, all cache calls are silent no-ops. The app works normally; you just lose the speedup.
- **Lifecycle:** `init_redis()` / `close_redis()` run in the FastAPI lifespan (`main.py`).
- **Where it's used:** currently only the leaderboard endpoint (`GET /events/{id}/projects?leaderboard=true`) with a 30s TTL. Key format: `leaderboard:{event_id}`.
- **Invalidation pattern:** writes that change cached data delete the key explicitly (see `routers/events.py` — voting deletes `leaderboard:{event.id}` after commit). The 30s TTL is a safety net in case invalidation is missed.

When adding a new cached endpoint: pick a stable key prefix, keep TTL short (≤60s) unless the data is immutable, and delete the key from every write path that touches the underlying data.
